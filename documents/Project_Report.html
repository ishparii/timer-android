We started this project by learning and updating StopWatch example with learning state patterns at the beginning. We drew the dynamic state UML diagram for different states, under what circumstances and what the actions are invoked when the state changes. We created similar packages similar to StopWatch: android, common and model. For the testing, we separated android from java, and tested them separately. Inside java package, we also tested ClockModel, TimeModel and StateModel.

Something that is worthy mentioning is the fact that the use of repository is extremely difficult for this project for some reason, and every time when a team member commits and pushes some code into the repository, there are thousands of lines of system files being created, mostly binary, making it hard for the other members to read. This might partially, if not mostly, due to the use of virtual emulator on different individuals’ environments.

In terms of model and code, we created our state machine model first with initial actions, and then we also created a timer model facade that integrates StateModel, TimeModel and ClockModel together. Finally, we created a timer Adapter that coordinates View and TimerModelFacade and injected dependency into View. And without doubt, after all of the above steps, we tried to run the app, and the app crashed many times. It became effective finally after dynamic debugging and improvement. The relationship between model and code is that the model is used by the View with the code being adapter in between, in other words, the model has its own integrity, and it is not affected by View. The adapter connects model and view only to achieve the requirement.
